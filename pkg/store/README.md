# ZFS Storage and Replication Management (floatlab.store.zfs)

The floatlab store package provides the backbone of zfs replication for floatlab. This includes:
- Provisioning storage for new containers
- Provisioning storage for container volumes and mount points
- Adjusting ZFS parameters (block size, compression)
- Responding to ZFS Fault Conditions
- Maintaining ZFS Pools (pool scrubs, reporting ZED errors, adding disks and replacing faulted disks)
- Replicating running (master) container filesystems (recursively) to the secondary node
- Replication running (master) container filesystem to the backup node
- Creating and cleaning up scheduled snapshots (hourly/daily/weekly/monthly)
- Ad-hoc snapshots generated by users
- Responding to configuration changes, updating schedules, replication changes
- Performance Reporting (disk i/o latency, throughput, disk used (data + snapshots), free space, volume % used)

### New Volumes
- Each node can have one or more volumes, the first (main/primary) volume will always be called `floatlab` and will be used for the node configuration and storing user workloads.
- Filesystem `floatlab/floatlab-config/node-<name>` used for all local configuration for each node (replicated for all online nodes)
- Filesystem `floatlab/floatlab-stats` used to store all stat counters 
- Filesystem `floatlab/floatlab-logs` used to store all logs

### Provisioning
- Each container stack needs a dedicates ZFS filesystem that will contain all the volumes and bind mount as child ZFS filesystems, and the configuration for the stack
- Configurable Block Size (4-8k for databases, 32k default, 128k for blob storage)
- Configurable Compression (none, lz4 (default), gzip, zstd (for larger block sizes))
- Configurable Storage Quotas (Unlimited by default) per Stack, and per Mount/Volume
- ZFS Paths should match:
  - Container Stack Configuration /<storage-pool>/<stack-name>
  - Bind Mounts <storage-pool>/<stack-name>/<container-name>/<volume-name>
  - Volumes <storage-pool>/<stack-name>/<container-name>/vol-<volume-name>
  - Example Tree:
    - floatlab/my-app/docker-compose.yml
    - floatlab/my-app/db/data
    - floatlab/my-app/db/config
    - floatlab/my-app/web/data
    - floatlab/my-app/web/vol-npm-cache
 
 
### ZFS Faults
- This module will be responsible for monitoring ZFS for fault events (failed disk, read/write/checksum errors, etc)
- Theses error will be reported to the floatlab.notify service

### ZFS Statistics
- This module will be responsible for monitoring ZFS for statistics (disk i/o latency, throughput, disk used (data + snapshots), free space, volume % used)
- These statistics will be reported to the floatlab.stats service
- Statistics will be collected every 5 minutes and will be stored in the `floatlab/floatlab-stats` filesystem 
- Statistics will be used to generate reports and alerts for the floatlab.notify service based on disk usage

### ZFS Logs
- This module will be responsible for monitoring all `zfs send` and `zfs receive` replication jobs, and reporting success/etc to the logging service

### ZFS Snapshots
- This module will be responsible for creating and cleaning up scheduled snapshots (hourly/daily/weekly/monthly)
- Ad-hoc snapshots generated by users will be stored with the name `user-<date>-<time>-<name>`
- Scheduled snapshots will be stored with the name `flsnap-<date>-<time>-<type)` where type is hourly, daily, weekly, monthly, etc
- Replication snapshots will be named `fsrepl-<seq>-<date>-<time>`, once this has been replicated to the slave node, the previous snapshot will be removed

### ZFS Re-configuration
- Subscribe to the floatlab.config service for changes to any replication  slave, backup target, and snapshot schedule
